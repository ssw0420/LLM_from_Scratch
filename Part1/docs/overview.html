Step 3 바이그램 모델

4단계 발전: gen → word → bos_eos → verify

C ▶

gen - 고정 길이 글자 스트림

직전 1글자 → 다음 글자 예측. 200자 연속 생성 (단어 경계 없음)

word - EOS 토큰 도입

CP_END(EOS) 토큰으로 단어 단위 생성 가능. freq_start[] 별도 배열로 시작 분포 관리

bos_eos - BOS/EOS 통합

CP_BOS/CP_EOS를 BigramTable에 통합 → 생성 루프 단순화 + CSV 출력 (131,853행)

verify - 사전 대조 검증

42만 단어 해시셋 → 생성 단어와 완전 일치 비교

사전 적중률: 30개 중 ~50% — 짧은 단어는 잘 맞지만 길어질수록 하락

Step 5 N-gram 문장 생성

소설 코퍼스 + 3/5/7-gram 비교

C ▶

사전(단어) → 소설 코퍼스(문장)로 확장. 데이터: 번역 SF 소설 35,034문장, 190만 글자.

모든 문자(공백, 쉼표, 마침표)를 동등한 토큰으로 처리. 자료구조를 N 가변으로 일반화.

N-gram컨텍스트 수생성 품질
3-gram37,921단어 경계 불안정, 문법 파괴
5-gram480,542어절 자연스러움, 절 간 연결 끊김
7-gram1,146,338원문급이지만 과적합(암기)
N-gram 근본 한계 확인:
N↑ → 품질↑ 이지만 원문 암기 (과적합)
N↓ → 장거리 의존성 불가
데이터 증가로도 해결 불가 → 신경망 전환 동기

Step 6 형태소 토큰화 + 형태소 N-gram

6-1. Kiwi 형태소 분석

Python ▶

글자 단위 → 형태소 단위로 토큰 크기 전환. kiwipiepy 라이브러리 사용.

"랄프는 울었습니다"
→ [랄프, 는, <SP>, 울, 었, 습니다]

<SP> 토큰: 원문의 공백 위치를 보존하여 N-gram에서 띄어쓰기도 학습

종성 자모 후처리: 크 ᆫ → 큰, 하 ᆯ → 할

1,476,464 토큰 (어휘 16,640개) — 글자 대비 시퀀스 길이 22% 감소

6-2. 토큰 ID 시퀀스

Python ▶

LLM은 문자열이 아닌 정수 ID를 입력받음. 이 구조를 도입:

[랄프, 는, <SP>, 울, 었, 습니다]
→ [360, 8, 3, 164, 6, 37]

특수 토큰: PAD(0), BOS(1), EOS(2), <SP>(3)

빈도순으로 ID 부여 — 자주 쓰는 토큰이 작은 ID를 받음.

Python ▶

같은 N에서 형태소 단위는 글자 대비 2~3배 넓은 범위를 커버:

구리를 고스란히 백 퍼센트 에너지로

6-4. 백오프(Backoff) 시연

Python ▶

백오프: 높은 N-gram에서 후보가 부족하면(과적합/암기) N을 낮춰서 후보를 늘리는 기법.

10-gram 후보 1개 → 암기! → N을 낮춤
 7-gram 후보 3개 → 부족  → N을 더 낮춤
 5-gram 후보 15개 → 충분! → 여기서 샘플링

백오프 없음 (10-gram 고정)

"돈을 주지 않으면 날이 새기 전에 너희들을 모두 죽이어버리겠다."
→ 원문 그대로 재생 (암기)

백오프 적용 (10→7→5→3)

"돈을 그만 두지 않았지만 말하었습니다."
→ 새로운 문장 생성
